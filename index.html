Pookeh — Competitive PvP Gameplay
About the Game

What Is Pookeh?

Pookeh (پوکه؛ مبارزان کمپانی) is a real-time online Hero Shooter set in a dystopian sci-fi future controlled by a powerful multinational corporation. Players choose heroes and compete in intense team battles — think Overwatch, built in Iran, for Iranian networks.

The game launched to 100,000+ registered players and won three awards at Iran's largest government-recognized game festival — the 9th Fajr International Game Festival (1402), which had 200+ submissions and returned after a 4-year hiatus.

My Contributions

What I Built

I was the lead gameplay and multiplayer developer. While the team handled art and design, I owned the entire technical infrastructure — from the moment players connect to the moment the match ends.

01
Full Multiplayer Stack
Built the complete online infrastructure using Photon PUN2 and Fusion: lobby system, matchmaking, real-time room management, and team assignment. Players go from menu to match in seconds.
02
Server-Side Hit Validation
All hit detection is processed server-side. No client can fake a kill or manipulate damage — critical for a competitive game where fairness is everything.
03
Lag Compensation System
Iranian networks are notoriously unreliable (150ms+ latency is common). I implemented a full lag compensation layer using Photon Fusion's tick-based rollback, making fights feel responsive and fair even on bad connections.
04
Tournament System
Designed and built end-to-end: bracket generation, match scheduling, result tracking, prize distribution, and a full UI — all synced with the multiplayer backend. Greenfielded from zero with no existing framework.
05
Core Combat Systems
Every piece of combat you see in the game: character controllers, weapon handling, reload mechanics, ability systems, hit detection, and all the feedback systems that make it feel good to play.
06
Spectator Mode
Built a full spectator implementation allowing eliminated players and observers to watch ongoing matches — a necessary feature for the competitive tournament scene.
07
Performance Optimization
Profiled the game using Unity's profiler, identified rendering and physics bottlenecks, and applied LOD, object pooling, and draw call batching — achieving a 20%+ FPS improvement across target hardware profiles.
Technical Challenges

Problems I Solved

The Problem
Iranian networks have notoriously high latency. Building a fair competitive shooter for this audience meant network lag was the #1 gameplay issue.
The Solution

Implemented lag compensation using Photon Fusion's tick-based system with server-side rollback and validation. Each client predicts locally while the server remains authoritative. Result: maximum 5% packet loss even on worst-case networks — a figure that's exceptional for the region's infrastructure.

The Problem
The tournament system was a greenfield feature — no schema, no bracket logic, no backend, no UI. Just a requirement: "players need to compete in tournaments."
The Solution

Designed the entire system from scratch: bracket generation algorithm, match scheduling tied to Photon room creation, server-synced result tracking, and prize distribution logic. The feature shipped on time and was directly tied to a measurable increase in daily active users and in-app purchase conversion.

The Problem
The target audience uses a wide range of mid-to-low-end Android hardware. The game felt inconsistent — some devices hit 60 FPS, others dropped below 30.
The Solution

Ran deep profiling sessions, found the main bottlenecks in rendering (overdraw, dynamic shadows) and physics (unnecessary collision checks). Applied LOD groups, GPU instancing, object pooling for projectiles, and draw call batching. Average FPS improved by 20%+ across target devices.

Everyone's Train Town — Full Gameplay Showcase
About the Game

What Is Train Town?

Everyone's Train Town is a mobile simulation builder where players construct and manage their own railway town — placing buildings, customizing layouts, running trains on tracks, and growing their in-game economy.

The game shipped to both iOS and Android as a cross-platform live title. The team of 8 included designers, artists, and QA — and I was the only engineer on the project. Every line of gameplay code is mine.

My Contributions

What I Built

I designed and built everything from scratch. The art team provided assets; I made them into a game.

01
Building & Customization System
Full snap-to-grid placement system with rotation, collision detection, upgrade paths, and visual feedback. Players can build and rearrange their town freely — with instant undo/redo and persistent saves.
02
Train Locomotion System
Trains follow custom spline-based tracks with physics-influenced speed, acceleration curves, and station-stopping logic. Multiple trains run simultaneously without collision or desync.
03
In-Game Economy
Coin earning, spending, pricing balance, and progression rewards tuned for a mobile-first loop. Includes production buildings, passive income, and timed rewards — all designed to keep players engaged.
04
Addressables + ScriptableObject Pipeline
The most impactful architectural decision on the project: a content management system that let the art team add new train types (models, stats, pricing) via a shared spreadsheet — without requiring a new app build.
05
SDK Integrations
Gmail OAuth for cross-platform identity (players keep their progress across devices) and Zepeto SDK for character sync — enabling social avatar features tied to the game's community platform.
06
All UI Systems
Every screen, menu, HUD element, shop interface, and popup — built from scratch. Optimized for mobile touch input with consistent visual feedback and smooth transitions.
Technical Challenges

Problems I Solved

The Problem
The art team needed to add new train types frequently, but every content change required a full app build, QA cycle, and store submission — a 1–2 week delay per update.
The Solution

Designed a live-update pipeline using Unity Addressables for remote asset loading and ScriptableObjects as data containers synced via a shared spreadsheet. The art team updates the sheet and publishes Addressables — new train types appear in the live game with zero app rebuild required. Eliminated week-long release cycles for content updates entirely.

The Problem
Being the sole engineer meant enormous scope: economy, locomotion, building systems, UI, SDKs — all simultaneously, with a real team and real users waiting.
The Solution

Prioritized modular, decoupled architecture from day one. Each system (economy, building, locomotion) operates independently with clean interfaces — so adding a feature never breaks an existing one. This made it possible to iterate quickly without accumulating debt. Shipped on schedule.

Dobeil | MOBA Gameplay Demo — Server-Authoritative Multiplayer Prototype
About the Project

What Is Dobeil?

Dobeil is a fully functional MOBA prototype — heroes, lanes, creeps, towers, team objectives — built entirely solo over three years. No team, no networking framework, no shortcuts.

This project exists because I wanted to understand how competitive multiplayer games really work at the infrastructure level. Every system was built from first principles: the networking layer, the server authority model, the game simulation, all of it. The result is a near-complete playable demo that demonstrates the same architecture patterns used by professional game studios.

My Contributions

What I Built

Everything. No co-developers, no networking library, no game framework beyond Unity and C#.

01
Custom TCP/UDP Networking Layer
Built from raw C# sockets: packet serialization, reliable and unreliable channels, connection management, and reconnection logic. No third-party networking library — written entirely from scratch.
02
Server-Authoritative Architecture
All gameplay logic — movement, combat, ability resolution, creep behavior — is computed on the server. Clients are intentionally "dumb": they send inputs and receive world state. Zero client-side cheat vectors.
03
Input-Based Networking Model
Clients transmit discrete inputs (move, attack, ability) rather than positions. The server simulates the game tick by tick. This is the exact same networking model used in Valve's Source Engine and most competitive PC games.
04
Client-Side Prediction & Reconciliation
Clients predict their own hero's movement locally (so controls feel instant), then reconcile with server corrections when they arrive. Latency becomes invisible for the most common action: moving.
05
Creep AI System
MOBA-style lane creeps with spawn logic, pathfinding along pre-defined lanes, aggro range detection, combat state machine, and death/respawn sequences — all simulated server-side.
06
Tower Combat & Hero Abilities
Tower targeting, range detection, attack cooldowns, and destruction sequences. Hero ability systems with cooldowns, hitboxes, and projectile logic — all validated server-side for fairness and cheat prevention.
Technical Challenges

Problems I Solved

The Problem
MOBA games require many entities (heroes, creeps, towers, projectiles) to be perfectly synchronized across all clients. Any desync makes the game unplayable.
The Solution

Full server-authoritative state sync: every game tick, the server computes the authoritative world state and broadcasts delta updates to each client. Clients never calculate outcomes — only the server does. State is fully deterministic; if two clients receive the same inputs, they produce identical results.

The Problem
Raw TCP/UDP sockets require manually handling packet fragmentation, out-of-order delivery, dropped connections, and message reliability — problems that networking libraries normally solve for you.
The Solution

Built a lightweight reliability layer on top of UDP: gameplay-critical messages (ability use, death events) get acknowledged and retransmitted if dropped. Position updates use unreliable UDP — stale positions are harmless since the next update arrives quickly. This hybrid approach matches what Valve and Riot use in production.

DinoWay — Casual Mobile Gameplay
About the Game

What Is DinoWay?

DinoWay is a casual mobile puzzle game with a unique twist: instead of traditional waypoint markers or minimaps, players navigate by following dynamically generated footsteps left by the character. The footsteps are your guide — follow them to progress.

The game was a side project born from a simple question: "What if the navigation system was part of the game world instead of layered on top of it?" Three friends, three months, one published app.

My Contributions

What I Built

01
Core Mechanic Design
The procedural footstep guidance system — the idea, the design, and the implementation. Players follow footsteps instead of arrows or minimaps, keeping navigation inside the game world and feeling natural.
02
Procedural Footstep System
A real-time path generator that creates footstep sequences ahead of the player, adapting to the current level layout. Footsteps spawn, animate in, and fade out as the player moves through the environment.
03
Player Controls
Mobile touch controls tuned for one-handed play. Clear input response with satisfying physics — the controls needed to feel good since gameplay is simple, making polish the differentiator.
04
Feedback Systems
Particle effects, sound trigger integration, and animation hooks for the footstep reveal, player actions, and puzzle completion. The feel of the game — not just the function.
Technical Challenges

Problems I Solved

The Problem
Casual games lose players in the first 30 seconds when navigation is confusing. Traditional waypoint arrows feel clinical and break immersion in a character-driven puzzle game.
The Solution

Designed the procedural footstep system as a diegetic navigation tool — players follow their character's own footprints. It feels intuitive because it mirrors real-world logic. No UI overlays, no arrows, no minimap. The game world is the interface. This also doubles as a puzzle mechanic: sometimes the footstep path isn't the obvious route.

The Problem
3 months is not a lot of time to design, build, test, and publish a game — especially as a side project alongside a full-time job.
The Solution

Kept scope ruthlessly controlled: one core mechanic, polished well, rather than many mechanics done half-heartedly. Built modular systems so levels could be designed by the team without requiring new code. Published on schedule.
